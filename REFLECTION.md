# Reflection

## Development process and key architectural decisions

I had a hard ~4-hour budget, so the first decision was about where time would actually pay off: the data-grounding layer, not UI polish. I connected to the Snowflake Marketplace dataset first and inspected the real schema before designing anything, rather than assuming a shape — that surfaced two facts that drove the whole design:

1. The dataset is ~30 wide tables per year plus a metadata table describing 8,000+ individual columns, with no single small table that says "population by state." There's no way to hardcode the relevant columns for every plausible question, so I built an **agentic tool-use loop**: the model gets `search_census_tables`, `get_table_fields`, and `run_sql`, plus a primer on the schema's *shape* (naming conventions, the geography join), and explores the actual metadata at query time instead of relying on a prompt that enumerates every column.
2. The geography join has a real gotcha: `METADATA_CBG_FIPS_CODES` is keyed one-row-per-county, and naively joining a data table to it on state alone fans out and silently inflates sums (I caught this live — it inflated California's population by ~58x before I fixed the join condition). This is exactly the kind of bug that produces a confident, wrong-but-plausible number rather than an obvious crash, which is the failure mode the assignment cares most about. I encoded the correct join pattern directly into the agent's system prompt so it can't rediscover the bug at runtime.

Architecturally, I split guardrails from the main agent into a separate, cheaper model call specifically so off-topic/inappropriate questions fail fast instead of spending tool-use turns and Snowflake round-trips on a question that was never going to be answered — this also keeps the 60-second budget safer on adversarial input. I structurally prevented hallucination rather than just prompting against it: the model can only produce a final answer after tool results are already in its context, so there's no code path where it answers without having queried Snowflake first.

I used Claude (via this session) throughout — for design discussion, for writing all the code, and notably it caught the JSON-in-markdown-fence guardrail parsing bug during live testing (the classifier model wrapped its JSON output in ` ```json ` fences despite instructions not to; I added defensive stripping and a regression test). That live-testing loop — running real queries against real Snowflake data rather than trusting the design on paper — is what surfaced both real bugs (the join fan-out and the markdown fence) in the time available.

## What I'd improve with more time

- **Logging granularity was initially too coarse to debug with.** Early structured logging showed *that* an answer was slow or fast, but not *why* -- whether it came from the LLM's own conversation memory, a cache hit, or a fresh Snowflake round-trip, you had to infer from which log lines were absent. Fixed: every tool call now logs explicitly (`Tool call: name(args)`), `schema_tools` and `snowflake_client` log "Cache hit" vs "Cache miss" on every lookup, and each turn ends with an explicit summary (`Turn complete in Xs via LLM/conversation memory (0 tool calls)` or `via N tool call(s), M error(s)`). This isn't cosmetic: building this out surfaced a real, fixable inefficiency immediately -- the model's first guess for a state filter sometimes used the full state name (`'Colorado'`) against a column that only holds 2-letter abbreviations (`STATE = 'CO'`), which doesn't error, it just silently returns `NULL` from the `SUM`, costing 2-3 extra self-correction tool calls before it found the right query. Fixed by stating the column's actual format explicitly in the schema primer. Verified live with two separate questions, not a before/after of the identical question: Colorado, tested before the fix, took 5-6 tool calls (~20-23s) across two runs; Wyoming, tested fresh after the fix, took 3 tool calls (~15s) and got the right SQL on the first attempt.
- **True token streaming.** The UI shows step-by-step progress (`run_agent_turn` takes an `on_progress` callback that reports "Searching tables for...", "Querying Snowflake...", etc. as each tool call happens, added after re-testing against the assignment's latency-UX tip specifically), but the final synthesized answer still renders all at once rather than token-by-token. With more time I'd stream the final completion call too.
- **Concurrent-user safety.** The Snowflake connection is a single module-level connection reused across requests (with a lock around connection creation, not around query execution). For a single demo user this is fine; under concurrent load on Streamlit Community Cloud (which can serve multiple sessions from one process) this is a real bottleneck/race risk, especially since I set the statement timeout via `ALTER SESSION` rather than a per-statement parameter. I'd move to a connection pool or per-session connections.
- **City/place-level geography.** The dataset has no city table; I could improve coverage by joining in a separate, free FIPS-to-place crosswalk and clearly labeling derived/approximate results, rather than just declining.
- **Statistical correctness of medians.** Median household income "by state" is currently computed by averaging block-group-level medians, which I disclose in the answer but isn't the methodologically correct way to estimate a population median (population-weighted aggregation or using a state-level ACS table directly would be more correct, if one exists in this dataset — I didn't have time to check).
- **Observability.** ~~No structured logging~~ -- addressed: `app.py` and `agent/snowflake_client.py` now log each question, the guardrail verdict, generated SQL with success/failure and latency, and any exception with a traceback, all to stdout (which Streamlit Cloud's log viewer captures natively, no new infrastructure needed). The Snowflake connector's own very verbose connection-handshake logging is suppressed to WARNING so it doesn't bury the actual application logs. Still missing for a real handoff: this isn't queryable/aggregatable (it's just a stream of text), so I'd want it shipped to a real log/metrics platform if this needed to scale.
- **Connection resilience.** Originally, if the Snowflake connection ever went stale mid-process (the exact kind of thing we saw live -- a query failed, then immediately succeeded on retry, consistent with the trial warehouse's auto-suspend/resume), there was no recovery path: every subsequent query would keep failing against the same dead connection until the whole app restarted. Fixed in `run_select`: a SQL-level error (bad column, bad syntax) still fails immediately, since retrying an actually-wrong query wouldn't help -- but anything else (connection drop, timeout) now forces a fresh connection and retries exactly once before giving up. This is a direct, evidence-based fix for the one real transient failure we observed, not a hypothetical.
- **Input length cap.** There was no limit on message length, so a single very large pasted message could burn a disproportionate number of tokens (cost) in one request. `app.py` now rejects anything over 500 characters before it reaches the guardrail or agent at all.
- **Conversation persistence.** Multi-turn memory works by keeping the full message history in Streamlit's `session_state` and resending it to Claude on every turn (`app.py`) -- this satisfies "preserve context across multiple turns" within a session, but it's in-memory only and scoped to one browser tab. Refreshing the page, closing the tab, or returning later starts a new, empty conversation; nothing is persisted to a database. For a real handoff I'd add a lightweight persistence layer (even just a session ID + a small table) if cross-visit memory mattered.
- **Token growth was a real, caught gap, not hypothetical.** Resending the full history every turn means a conversation's cost/latency grows with length -- and worse than it looks, since every tool call (search/lookup/SQL) within a turn also gets added to what's resent, not just one entry per question. A few tool-heavy questions could balloon context much faster than the 30-question session cap implies. Fixed with `agent_loop.trim_history`: it keeps only the most recent 8 user turns (with their full tool-use trace) and drops older ones, cutting only at clean turn boundaries -- it specifically never cuts between a `tool_use` block and its required `tool_result`, which the Anthropic API would reject as malformed. Verified with a 10-turn simulated conversation: context size stays flat (~29 messages) once trimming kicks in at turn 9, instead of growing unbounded. The UI shows a one-time note when older context actually gets dropped, so the user isn't confused by a follow-up to something from much earlier in the chat silently failing.
- **No caching, originally.** Before this was added, every question -- even an exact repeat of one asked a minute earlier, by the same or a different user -- re-ran schema lookups and SQL from scratch. Since this is read-only, immutable historical data, caching successful results is always safe. Added: a cache on the two schema-lookup tools (`search_census_tables`, `get_table_fields` in `schema_tools.py`), and one on `run_select` (`agent/snowflake_client.py`) keyed by the exact post-validation SQL string. (Originally the schema-lookup tools used `functools.lru_cache`, but I switched them to explicit checks once I added cache hit/miss logging -- `lru_cache` doesn't expose whether a given call was a hit or a miss, which made the logs ambiguous.) The one subtlety, true of all three: **errors are never cached** -- only successful results are, because we'd already seen a transient Snowflake warehouse-cold-start failure during testing; caching that would have made the failure permanent for the rest of the process's life instead of self-healing on retry. Verified live: an identical repeated question dropped from 23.2s to 12.6s (same correct answer both times) -- the remaining latency is Claude's own reasoning calls, which aren't cacheable since each is an independent LLM invocation.
- **Cache size limits were inconsistent until directly asked about them.** Only the `run_select` cache had a bound (256 entries, FIFO eviction); the two schema-lookup caches were plain dicts with no limit at all, an inconsistency that survived several rounds of review because it never came up explicitly. Low real-world risk here (the schema caches are naturally bounded by the dataset itself -- at most ~150 distinct table codes), but not *actually* bounded, so a long-running process with many distinct keyword searches would have grown that cache forever. Fixed properly rather than patched in place: extracted the FIFO-eviction logic that `run_select` already had into a small shared `BoundedCache` class (`agent/cache_utils.py`), and pointed all three caches at it, so the bound and the eviction behavior are identical everywhere instead of one bespoke implementation and two unguarded dicts. Added 6 direct unit tests for the class itself (`tests/test_cache_utils.py`), including one that inserts 100 entries into a 5-entry cache and asserts it never exceeds 5.
- **Rate limiting / cost protection.** ~~The app is behind a shared password but has no per-user rate limit~~ -- addressed: `app.py` now caps each browser session at 30 questions and enforces a 3-second minimum between messages, and a Snowflake Resource Monitor (`CENSUS_CHAT_AGENT_BUDGET`, 10 credit/month quota, auto-suspends the warehouse at 100%) caps worst-case account-wide spend regardless of what the app does. The session cap is in-memory only (resets if Streamlit restarts the process), and there's still no equivalent hard cap on the Anthropic side beyond whatever billing limit is set directly on that account -- with more time I'd add that and move the per-session limit to something that survives a process restart.

## Edge cases / failure modes identified but not fully addressed

Verified live and working correctly (not just assumed from the prompt): ambiguous geography ("Washington" -> picks the state, flags DC as the alternative), conflicting/contradictory geography ("Austin, California" -> correctly identified it as a real small place in Contra Costa County, distinct from Austin, TX, and offered both), underspecified queries (median income with no location -> defaults to nationwide and says so), years outside 2019-2020 (declines rather than substituting silently), and a topic the dataset doesn't cover at all (crime rate -> declines and suggests real sources). Adversarial phrasing (roleplay jailbreak, embedded "ignore your instructions") is caught by the guardrail classifier.

Not fully addressed:
- **Very broad questions** ("tell me everything about Texas") could plausibly cause the agent to run many tool calls and approach the 8-iteration/45-second budget; I tested complex-but-bounded questions (a two-state comparison, ~28s) but not a deliberately unbounded one.
- **Schema-table topics with no clean match** (e.g. a real but obscure ACS topic not surfaced well by simple `ILIKE` keyword search in `search_census_tables`) could cause the agent to miss an answerable question and incorrectly call it out of scope. The search is keyword/ILIKE-based, not semantic, so phrasing that doesn't share words with the table catalog could fail silently into a false "can't answer."
- **Concurrent users on the deployed instance** — see "connection safety" above; I have not load-tested this.
- **Observed once on the deployed app, not reproducible locally**: a request hit the max-iteration fallback ("I wasn't able to settle on an answer...") on the live Streamlit Cloud instance, but succeeded immediately on retry, and 3/3 identical local runs against the same live Snowflake account succeeded normally. The likely cause is Snowflake trial-warehouse auto-suspend: the first query after a period of inactivity has to resume the warehouse, which can take 10-20s and could eat into a tool call's budget, causing a slow/failed attempt that costs an iteration. The connection-retry fix above (see "Connection resilience") addresses the most likely underlying cause -- a stale/failed connection now gets one automatic retry against a fresh connection instead of just failing -- but I haven't been able to force-reproduce the original cold-start scenario to confirm it's fully resolved, since it depends on Snowflake's own warehouse suspend timing.

After the initial build, I deliberately ran a batch of edge cases the assignment calls out by name (ambiguous geography, a topic the dataset doesn't cover, an underspecified question, an unavailable year, and adversarial/jailbreak phrasing) rather than treating the happy-path tests as sufficient. Most behaved well -- e.g. asking for "the population of Washington" correctly defaulted to the state while flagging the DC ambiguity, and asking about Texas "crime rate" correctly said that's outside the ACS dataset rather than guessing. But that pass turned up a real security gap: the SQL-safety check only validated *that* a statement was a SELECT, not *which* database it could target. I confirmed live that this Snowflake role can read `SNOWFLAKE.ACCOUNT_USAGE.LOGIN_HISTORY` (real login events, including client IPs) -- data with nothing to do with the census dataset. The guardrail classifier already catches the obvious phrasing of an attempt to pull that data, but defense in depth means the SQL layer itself shouldn't trust the model's good behavior, so I added an explicit check rejecting any fully-qualified reference to a database other than the target one, independent of guardrail/prompt behavior, and added regression tests for it (`tests/test_sql_safety.py`). This is the same pattern as the geography join bug: a structural gap that only surfaces by actually attacking your own system, not by reading the code.

## Testing approach and tradeoffs

I wrote 53 unit tests covering six areas: SQL safety validation (rejects DDL/DML/multi-statement input, cross-database references, enforces row limits), the guardrail classifier (all three verdicts, fail-open behavior on malformed model output, the markdown-fence bug found during live testing, and follow-up-context handling), the agent loop's control flow (tool dispatch, the max-iteration fallback, the soft-deadline fallback, progress-callback reporting, exception containment, history trimming without breaking tool_use/tool_result pairing), caching (repeated lookups served from cache, errors never cached), the shared `BoundedCache` eviction logic itself (e.g. inserting 100 entries into a 5-entry cache and asserting it never exceeds 5), and connection resilience (SQL errors fail immediately, connection errors retry once). Snowflake and Anthropic are mocked in all of these, so the suite runs fast and without live credentials -- but mocked data can only test *control flow*, not whether the actual SQL produces the right number.

To close that gap I added `tests/test_golden_data.py`: 6 tests that hit live Snowflake and check real aggregation queries against the **official 2020 Decennial Census population** for 5 states (CA, TX, FL, NY, OH), independent ground truth rather than a number derived from our own code, with a 10% tolerance to allow for expected ACS-survey-vs-decennial-count variance. One of the six is a direct proof these tests have teeth: it deliberately re-runs the original buggy state-only join and asserts the result is wildly wrong (off by >2x) while the fixed join is within tolerance -- i.e. this exact test suite would have caught the bug that manual testing caught by luck. The suite skips automatically (not fails) when `SNOWFLAKE_ACCOUNT` isn't set, so it doesn't break CI environments without secrets.

What I'd still add with more time: a similar golden set that goes through the *full agent* (LLM + tools), not just hand-written SQL, to catch regressions in how the model translates a question into a query -- e.g. "what's California's population" → extract the number from the answer text → compare to the same ground truth. I didn't build that out because it's slower and costs API calls on every run, and the manual edge-case testing already covered a wide range of question phrasings; the SQL-level golden test was the higher-value, lower-cost piece given the time left.
