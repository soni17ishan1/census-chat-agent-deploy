# Reflection

## Development process and key architectural decisions

I worked under a tight, self-imposed time budget well inside the assignment's 24-hour window, so the first decision was about where time would actually pay off: getting the data layer right, not polishing the interface. Before designing anything, I connected to the real Snowflake dataset and looked at what was actually there, rather than guessing its shape. That look surfaced two facts that drove the whole design:

1. The dataset has roughly 30 wide tables per year, plus a separate reference table describing over 8,000 individual columns, and there's no single small table that just says "population by state." There was no realistic way to hardcode the relevant columns for every possible question, so I built the agent as something that explores the data on its own at question time — it can search the table catalog by topic, look up exactly what a table's columns mean, and then run a query — rather than relying on a prompt that tries to list every column up front.

2. The way the data is organized by geography has a real trap in it. Each small neighborhood-level row only identifies its location with a numeric code, and there's a separate lookup table, with one row per county, used to translate that code into a real place name. If you match rows to that lookup table using only the state part of the code and forget to also match the county part, every neighborhood ends up incorrectly matched against every county in its state at once. I hit this live: it inflated California's population by about 58 times before I found and fixed it. This is exactly the kind of mistake that produces a confident, plausible-looking wrong answer instead of an obvious crash — which is the failure mode this assignment is most worried about. I wrote the correct way to do that match directly into the instructions the model follows, so it can't reintroduce the same mistake later.

Architecturally, I split the "is this question even appropriate to answer" check into its own separate, fast pass, before the main agent ever gets involved. That way an off-topic or inappropriate question gets turned away immediately, instead of burning several rounds of database lookups on something that was never going to be answered — which also keeps response time safer under the 60-second limit when someone sends something adversarial. I also made it structurally impossible for the model to answer with a number it didn't actually get from the database: it's only ever allowed to write its final answer after it has already seen real results come back, so there's no path through the code where it can just make something up.

I used Claude throughout this whole process — for talking through the design, for writing all of the code, and for catching a real bug live during testing: the small classification step that decides whether a question is on-topic would occasionally wrap its answer in extra formatting it had been told not to use, which broke the code trying to read that answer. I added a fix that strips that formatting defensively, plus a test that locks the fix in. That pattern — actually running real questions against the real data instead of just trusting that the design looked right on paper — is what caught both that bug and the geography mistake above, in the time available.

## Gaps found during testing and fixed along the way

These were all real problems at some point during the build, not hypothetical — each one was found through actually testing the running system, then fixed and re-verified. Listed here, separately from genuinely outstanding work below, so it's clear these are resolved, not still pending.

- **No visibility into what the app was doing.** Originally there was no record of what happened for a given question, so debugging meant guessing. I added logging for every question, the topic-check result, every database query with its timing and outcome, and full detail on any failure — all written somewhere the hosting platform already displays, so no new infrastructure was needed. Once that was in place, it immediately wasn't detailed enough on its own: it showed *that* an answer was slow, not *why*. I made it log explicitly which of three things happened — answered from memory, from a saved result, or from a fresh database lookup — and that finer detail directly revealed a real, fixable problem: the model would sometimes guess a state's full name where the data only stores a two-letter code, which doesn't error, it just quietly returns nothing, costing 2-3 wasted slower attempts before stumbling onto the right format. Fixed by stating the correct format up front in the model's instructions; verified live, a comparable question dropped from 5-6 of these wasted attempts down to 3, succeeding immediately.
- **A failed database connection had no way to recover on its own.** If the connection to the database ever went stale mid-session — and we did see this happen live once, where a question failed and then the identical question succeeded moments later — every following question would have kept failing against that same broken connection until the whole app restarted. Fixed so a genuine query mistake still fails immediately (retrying it wouldn't help), but anything that looks like a connection problem now automatically opens a fresh connection and tries once more before giving up.
- **No limit on how long a single message could be.** A single very large pasted message could have driven up cost disproportionately in one request. Fixed: the app now rejects anything over 500 characters before it's even looked at.
- **Long conversations were getting more expensive and slower, not just in theory.** Every previous question and every database lookup within it gets resent to the model on every new question, so a long conversation — especially one with several data-heavy questions — cost and took more time than the question count alone would suggest. Fixed by keeping only the most recent several exchanges and dropping older ones cleanly, careful never to cut in the middle of a single exchange and leave it half-recorded. Tested with a simulated ten-question conversation: the amount being resent stopped growing once the limit kicked in, instead of increasing without bound.
- **Nothing was being remembered between repeat questions.** Every question — even an exact repeat of one asked a minute earlier, by anyone — re-ran every lookup and every database query from scratch. Since the underlying data never changes, remembering a successful result and reusing it is always safe. Added that, with one important rule: a failed attempt is never remembered, only a successful one, because a temporary database hiccup had already happened once during testing, and remembering a failure would have made it permanent until restart instead of letting it recover on its own. Verified live: asking the identical question twice dropped from about 23 seconds to about 13 (the remaining time is the model's own thinking, which can't be skipped this way).
- **The amount being remembered wasn't actually limited everywhere.** Only one of the three places that remember results had a cap on how much it could hold; the other two had none at all — low real-world risk given how few distinct topics this dataset has, but not an intentional guarantee. Fixed properly rather than patched in place: pulled the size-limiting logic into one shared, tested piece of code, and pointed all three at it, so the limit and the eviction behavior are identical everywhere.
- **A leaked password had no limit on what it could cost.** Added two independent protections: the app itself caps each conversation at 30 questions with a required 3-second pause between each, and — separately, directly on the database account, not in the app's code — a hard cap of 10 credits per month that automatically cuts off database compute once reached, regardless of what the app does.

## What I'd still improve with more time

These are genuinely unresolved, not yet attempted:

- **The final answer isn't streamed word-by-word.** The interface does show live step-by-step status while the agent is working ("looking up tables...", "querying the database..."), but the finished answer still appears all at once rather than typed out gradually.
- **Everything currently shares one database connection.** Fine for a single person trying the demo, but under real concurrent traffic, multiple users could be sharing that one connection at the same time — a real bottleneck and a possible source of subtle bugs. I'd move to a proper pool of connections, or one per user session.
- **No way to answer questions about specific cities.** The dataset simply isn't organized by city name, only by small numbered neighborhoods. I could improve this by joining in a separate, free reference table that maps neighborhoods to city names and clearly labeling the result as approximate, rather than declining city questions outright as it does now.
- **Conversation memory only lasts for the current visit.** The system remembers earlier questions in the same conversation so follow-ups work correctly, but that memory lives only in the current browser tab; refreshing the page or coming back later starts over with nothing remembered. I'd add a lightweight way to save and restore a conversation if remembering across visits mattered.
- **An equivalent hard spending cap on the AI model's side.** The database has a hard monthly spending cap enforced by the platform itself; the AI model side only has the app's own session limit, not an equivalent platform-enforced cap. That has to be set directly on that account, which the app's code can't do on its own.

## Edge cases and failure modes

Verified live and working correctly, not just assumed: a state name that's also ambiguous with another well-known place picks the more likely interpretation and says so; a place name that could plausibly mean two different real locations gets explicitly disambiguated rather than guessed at; a question with no location specified at all defaults sensibly and says what it assumed; a year outside the range this data covers is declined rather than silently answered with the nearest available year; and a topic the dataset simply doesn't track is declined with a suggestion of where to actually find it. Attempts to manipulate the system into ignoring its instructions or revealing internal details are caught by the topic-check step before they reach anything else.

Also found, while deliberately testing for problems rather than just the happy path: a real security gap, where the check on generated database queries only confirmed they were read-only, not which part of the database they were allowed to read from. I confirmed live that the account this app uses can technically read its own internal usage records — login history, including IP addresses — which has nothing to do with the census data. The topic-check step already catches an obvious attempt to ask for that directly, but a system shouldn't rely on one layer catching everything, so I added a second, independent check at the database-query level itself that blocks any reference outside the intended dataset, no matter how the request is worded, plus tests that lock that in. This is the same category of issue as the geography mistake described above: something that only shows up when you deliberately try to break your own system, not from reading the code and assuming it's fine.

Not fully addressed:
- **Very broad, open-ended questions.** A request like "tell me everything about a state" could plausibly cause the agent to run many lookups in a row and approach its step/time limits. I tested complex-but-bounded questions (a side-by-side comparison between two states), but not a deliberately unbounded one.
- **Obscure topics phrased very differently from how the dataset describes them.** The topic search works by matching words, not meaning, so a real but obscure subject phrased in unfamiliar terms could be missed and incorrectly treated as unavailable.
- **Multiple people using the deployed app at the same time.** Related to the shared-connection limitation above; I haven't load-tested this.
- **One unexplained slow/failed response, seen once on the live deployment but not reproducible locally.** A question failed with a generic "couldn't find an answer" message on the hosted version, then succeeded immediately on retry, and three identical attempts run directly against the same live database all succeeded normally. The most likely explanation is that the database's compute had been paused after a period of inactivity and needed a few extra seconds to resume, eating into the time budget for that one attempt. The automatic-retry fix described above should address the most likely cause, but I haven't been able to deliberately force that exact pause-and-resume scenario to confirm it's fully resolved, since it depends on timing controlled by the database platform itself, not this code.

## Testing approach and tradeoffs

The test suite has 53 tests that don't touch the real database or the real AI model at all — they use stand-ins for both, so they run instantly and don't need any real credentials. They cover: the database-query safety checks (rejecting anything that isn't a plain read, rejecting attempts to reach outside the intended dataset, enforcing result-size limits); the topic-check step (all three possible outcomes, what happens if its response can't be understood, and that it correctly uses recent conversation for follow-up questions); the main agent's control flow (how it decides which tool to use, what happens when it runs out of attempts or time, and that trimming old conversation history never leaves a lookup half-recorded); the remembering/reuse behavior described above; and the connection-recovery behavior described above. Because these all use stand-ins instead of the real systems, they can confirm the logic behaves correctly, but they can't catch a case where the logic is fine but the actual data comes back wrong.

To close that gap, I added a separate set of 6 tests that do hit the real, live database, and check real results against the **official published 2020 population figures** for five states — an independently known correct answer, not something derived from this project's own code — allowing a 10% margin for the expected difference between a multi-year survey estimate and an exact once-a-decade count. One of those six tests exists specifically to prove the others have teeth: it deliberately re-runs the original broken version of the geography query and confirms the result really is wildly wrong, while the corrected version stays within the expected range — in other words, this is the exact test that would have caught the original bug automatically, rather than catching it by manually noticing an absurd number.

What I'd still add with more time: a similar check that goes through the full agent — asking it an actual question in plain language and confirming the number in its written answer matches the known-correct figure — rather than only testing the underlying query by itself. I didn't build that out because it's slower and uses real AI-model calls every time it runs, and the manual testing described throughout this document already covered a wide range of how people might actually phrase questions; checking the underlying queries directly was the more valuable use of the remaining time.
